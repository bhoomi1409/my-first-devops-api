💬 Capstone 1: Designing WhatsApp / Signal at 100M Users

Designing a real-time messaging gateway that supports 100 million Daily Active Users (DAU) requires moving away from traditional REST over HTTP and relational databases. Here we engineer the complete end-to-end architecture handling 350,000 messages/sec with sub-50ms global delivery.

2. Wire Protocol & Schema Design (Why JSON over REST is Malpractice)

Sending a JSON payload like {"sender": "Alice", "message": "Hey"} over HTTP/1.1 requires establishing a TCP connection, transmitting 500+ bytes of HTTP headers (User-Agent, Cookies, Host), parsing text formatting in user-space, and closing the socket. Doing this 350,000 times per second burns massive CPU serialization cycles and saturates network interface cards.

Instead, we establish a **Persistent TLS 1.3 WebSocket** upon app launch and transmit compact binary packets framed with Google Protocol Buffers:

// chat_protocol.proto — Compiled to zero-copy binary C++/Java/Rust structs syntax = "proto3"; package whatsapp.protocol; message ChatFrame { string message_id = 1; // 36-char UUIDv4 ensuring idempotency & deduplication string sender_id = 2; // Immutable user identifier string receiver_id = 3; // Target user or group channel ID bytes encrypted_payload = 4; // Signal Protocol E2EE ciphertext (Server cannot decrypt) int64 timestamp_ms = 5; // Client epoch timestamp enum FrameType { MESSAGE = 0; ACK_SERVER_RECEIVED = 1; // 1st Gray Checkmark ACK_DEVICE_DELIVERED = 2; // 2nd Gray Checkmark ACK_DEVICE_READ = 3; // Blue Checkmarks } FrameType type = 6; }

3. Step-by-Step Message Execution Trace (Alice $\rightarrow$ Bob)

  1. Socket Initialization: Alice opens the app. Her device initiates a TLS 1.3 handshake with **Gateway Pod #102**. Pod #102 registers Alice's live connection state in a high-speed **Redis Presence Cluster**: SET presence:alice "pod_102:10.0.4.15" EX 60 (heartbeat renewed every 45s).
  2. Ingress & Instant Server ACK: Alice sends a ChatFrame destined for Bob. Gateway Pod #102 parses the binary frame and **instantly returns an ACK_SERVER_RECEIVED frame back down Alice's socket**. Alice's UI immediately displays **1 gray checkmark**. Total latency: $<5\text{ms}$.
  3. Decoupled Persistence (Kafka $\rightarrow$ Cassandra): Gateway Pod #102 must never perform synchronous disk database writes while holding open sockets! It writes the raw frame asynchronously to an **Apache Kafka** topic partitioned by message_id.

    A pool of storage consumer workers reads from Kafka and appends the message sequentially into an **Apache Cassandra LSM-Tree wide-column cluster**.

  4. Why Cassandra over PostgreSQL? (Crucial Architectural Trade-off):
    Storage EngineWrite Latency at 350k req/sInternal Disk Page Mechanics
    PostgreSQL (B-Tree)Crashes / Multi-second timeoutsUpdating random B-Tree indexes across 10 Billion rows forces random read-modify-write disk jumps and page splits. Buffer pool contention causes fatal CPU lock freezes.
    Apache Cassandra (LSM-Tree)$\mathbf{1.2\text{ms}}$ deterministic commitWrites append strictly in memory (Memtable) and sequentially to an append-only disk CommitLog. When full, Memtables flush to immutable SSTables on disk with zero random seeks or page lock contention!
  5. Presence Lookup & Routing: A Message Dispatcher worker pulls the frame from Kafka and queries Redis: GET presence:bob.
    • Scenario A (Bob is Online on Gateway Pod #405): Redis returns Pod #405's internal IP. The Dispatcher publishes the binary frame over internal gRPC (or Redis Pub/Sub) directly to Pod #405. Pod #405 pushes the frame down Bob's open WebSocket. Bob's phone emits an ACK_DEVICE_DELIVERED frame back through the gateway network to Alice $\rightarrow$ Alice sees **2 gray checkmarks**. When Bob opens his screen, his device emits ACK_DEVICE_READ $\rightarrow$ Alice sees **2 blue checkmarks**.
    • Scenario B (Bob is Offline): Redis returns nil. The Dispatcher routes an encrypted push payload to Apple Push Notification Service (APNs) or Google FCM. When Bob turns on his phone 6 hours later, his app connects to a Gateway Pod and executes a delta sync: SELECT * FROM chat_messages WHERE receiver_id = 'bob' AND message_id > 'last_seen_uuid' against Cassandra.

4. Principal FDE Interview Dialogue Script

🎭 Live Interview Roleplay: Defending Your Architecture

Interviewer: "What happens if Gateway Pod #102 crashes abruptly while holding 500,000 active WebSocket connections?"
Candidate: "If Pod #102 loses power, 500,000 client sockets drop simultaneously. If those 500k phones immediately attempt reconnection at the exact same millisecond, they form a synchronized Thundering Herd that will overwhelm our DNS resolvers and remaining Gateway pods. To survive this, we embed two mandatory defenses in the mobile app firmware: first, **Exponential Backoff with Full Jitter** (`sleep = random(0, min(max_delay, base * 2^attempt))`), spreading reconnect attempts smoothly across a 90-second window. Second, our Anycast L4 load balancers route reconnecting clients evenly across the remaining 29 Gateway pods based on least connections."
Interviewer: "How do you handle group chats with 1,000 members without duplicating 1,000 database writes per message?"
Candidate: "For large groups, writing 1,000 individual copies of every message into Cassandra destroys disk bandwidth. Instead, we implement **Fan-Out on Read for Storage, and Fan-Out on Write for Delivery**. In Cassandra, we write exactly 1 copy of the message inside a `group_messages` table keyed by `(group_id, timestamp)`. For live delivery, the Message Router worker pulls the group's 1,000 member IDs from Redis cache, segments them into 20 batches of 50 users, and pushes binary frames concurrently to active Gateway pods. Offline users simply read from the shared `group_messages` partition upon reconnection."

🔗 Capstone 2: Designing an Enterprise URL Shortener at 10,000 req/sec

Building a high-throughput URL shortener like Bitly requires designing an atomic key generation pipeline that prevents database collisions while processing 10,000 short link creations per second and 100,000 redirects per second.

1. Mathematical Base62 Capacity Derivation

Why do industry shorteners use 7-character Base62 alphanumeric strings instead of MD5 hashes or UUIDs? A standard UUIDv4 is 36 characters long (`f81d4fae-7dec-11d0-a765-00a0c91e6bf6`)—too long to type or share on SMS. An MD5 hash is 32 hex digits.

Base62 consists of digits 0-9, lowercase a-z, and uppercase A-Z ($10 + 26 + 26 = 62\text{ characters}$). It omits URL-unsafe punctuation like +, /, and =.

If our enterprise platform creates 1,000 shortened URLs every single second ($31.5\text{ Billion/year}$), a 7-character Base62 string ($3.52\text{ Trillion}$ slots) will operate continuously at maximum load for **over 111 years before running out of unique combinations**!

2. The Key Generation Service (KGS) Architecture

A naive implementation generates a random 7-character string on the fly and executes a relational query: SELECT id FROM urls WHERE short_key = 'x9Kd2Pq'. If the key exists, it generates another and retries. Under 10,000 creates/sec, this read-before-write collision checking causes catastrophic database disk lock contention!

# The Offline Key Generation Service (KGS) Architecture [KGS Daemon Worker] ---> Pre-generates 100 Million unique 7-char Base62 keys ---> Inserts keys into Redis Set Pool: `available_keys` [API Creation Pod] ---> Receives POST /shorten {"long_url": "https://..."} ---> Executes atomic Redis SPOP `available_keys` (0.1ms latency!) ---> Writes (short_key, long_url) asynchronously to DB ---> Returns short link to user immediately. ZERO COLLISION QUERIES!

💳 Capstone 3: Designing an Immutable Stripe Payment Ledger

Financial transaction engines cannot tolerate lost updates, race conditions, or partial failures. Here we design a double-entry accounting ledger backed by PostgreSQL serializable isolation and Transactional Outbox CDC.

1. Double-Entry Bookkeeping SQL Schema

Never record bank account balances as a single mutable integer (`balance = balance - 50`). If a process crashes mid-update, auditing where funds went is impossible. We enforce immutable double-entry bookkeeping: every financial movement creates paired debit and credit rows inside a serializable ACID transaction.

-- Immutable Double-Entry Ledger Schema CREATE TABLE accounts ( account_id VARCHAR(64) PRIMARY KEY, currency CHAR(3) NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE ledger_entries ( entry_id UUID PRIMARY KEY, transaction_id UUID NOT NULL, -- Groups paired debits and credits account_id VARCHAR(64) REFERENCES accounts(account_id), amount DECIMAL(18, 4) NOT NULL, -- Never use FLOAT! Use exact decimal direction VARCHAR(6) NOT NULL CHECK (direction IN ('DEBIT', 'CREDIT')), created_at TIMESTAMPTZ DEFAULT NOW() ); -- Executing a $500 transfer from Alice to Bob: BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE; INSERT INTO ledger_entries (entry_id, transaction_id, account_id, amount, direction) VALUES ('e1', 'tx-9921', 'acc_alice', 500.00, 'DEBIT'), ('e2', 'tx-9921', 'acc_bob', 500.00, 'CREDIT'); -- Mandatory Accounting Invariant Assertion: -- SUM(DEBIT) minus SUM(CREDIT) inside any transaction group MUST EQUAL EXACTLY 0.00! COMMIT;

2. Transactional Outbox with Debezium CDC Tailing

When a payment completes, you must notify downstream systems (email receipt, fraud engine, analytics). If you execute sql_commit() and then call kafka.publish(), a server crash between line 1 and line 2 results in a completed payment with zero customer receipt or fulfillment!

To guarantee $100\%$ atomicity without dual-write bugs, we insert an event row into an local outbox_events SQL table inside the **exact same database transaction** that creates the ledger entries. An external Change Data Capture (CDC) daemon (**Debezium**) continuously tails the PostgreSQL Write-Ahead Log (WAL) and streams outbox records into Kafka with guaranteed at-least-once delivery.
