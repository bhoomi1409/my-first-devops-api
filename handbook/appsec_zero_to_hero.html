JSON Web Tokens (JWT) & Stateless Authentication

Unlike traditional stateful session IDs stored in database memory, a JWT is a self-contained cryptographic token verifying identity via an HMAC or RSA cryptographic signature.

💡 Security Gotcha Checklist:

  • Never store sensitive data (passwords, SSNs) in JWT payloads! The payload is merely Base64 encoded, NOT encrypted.
  • Always use strong secrets or RS256 asymmetric pairs. Weak secrets can be brute-forced offline in seconds using Hashcat.
  • Storage: Store tokens in HttpOnly cookies or memory to prevent XSS theft!

OAuth 2.0 & OpenID Connect (OIDC) Architecture

OAuth 2.0 is an Authorization protocol (delegated access without sharing passwords). OpenID Connect (OIDC) adds Authentication by issuing an ID Token proving user identity.

The Authorization Code Flow with PKCE (Proof Key for Code Exchange)

1. 🧑 Client generates random code_verifier and hashes it to create code_challenge.

2. ➔ Redirects browser to Auth Server (Google/Okta) with code_challenge.

3. 🧑 User logs in and approves consent screen.

4. ➔ Auth Server redirects back with temporary Authorization Code.

5. ➔ App sends Authorization Code + original secret code_verifier to token endpoint.

6. ✅ Auth Server verifies hash! Issues secure Access Token + ID Token!

Cross-Site Request Forgery (CSRF) Prevention

CSRF tricks a logged-in user's browser into auto-submitting cookies to perform state-changing requests on banking or API servers.

  • SameSite Cookie Attribute: Set cookies to SameSite=Strict or Lax to prevent browsers from sending cookies on cross-origin requests.
  • Anti-CSRF Synchronizer Tokens: Backend issues a random secret token verified on every POST/PUT request.

Cryptography Mastery: Symmetric vs. Asymmetric vs. Hashing

Understanding the exact cryptographic algorithms used in enterprise applications is critical:

Category Industry Standard Algorithms How It Works & Best Use Case
Symmetric Encryption
(Shared Secret Key)		 AES-GCM-256
ChaCha20-Poly1305		 Uses the exact same secret key to encrypt and decrypt data. Extremely fast! Used for encrypting database backups, cloud S3 buckets, and data at rest.
Asymmetric Encryption
(Public / Private Key Pair)		 RSA-4096
ECC / ED25519		 Uses a Public Key to encrypt or verify, and a secret Private Key to decrypt or sign. Used for HTTPS/TLS handshakes, SSH access, and signing RS256 JWT tokens.
Password Hashing
(One-Way Key Derivation)		 Argon2id
bcrypt / PBKDF2		 One-way mathematical functions that cannot be decrypted. Never use fast hashes like MD5 or SHA-256 for passwords! Modern GPUs can calculate billions of SHA-256 hashes per second. Argon2 and bcrypt intentionally consume RAM and CPU to make brute-forcing impossible.

CORS, SSRF, Rate Limiting & Zero-Trust Architecture

Security Topic Vulnerability / Risk Architectural Defense Standard
CORS
(Cross-Origin Resource Sharing)		 Setting header Access-Control-Allow-Origin: * alongside authentication credentials allows any malicious website on the internet to read your users' API responses! Specify exact trusted domain whitelists (e.g., https://app.company.com) and validate origin headers explicitly.
SSRF
(Server-Side Request Forgery)		 Attackers supply an internal URL (e.g., AWS Metadata IP http://169.254.169.254/latest/meta-data/) to a webhook endpoint, forcing your cloud server to leak internal IAM credentials. Implement strict network egress firewalls and block private IP ranges (10.0.0.0/8, 127.0.0.1) in input validators.
Rate Limiting & DDoS Defense Attackers flood authentication or GraphQL endpoints with millions of automated requests, causing database exhaustion. Implement Token Bucket or Sliding Window rate limiters at the API Gateway / Cloudflare level (e.g., max 5 login attempts per minute per IP).
