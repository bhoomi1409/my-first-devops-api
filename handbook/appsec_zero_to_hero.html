Level 1: What is an API Endpoint? & Attacker Reconnaissance

An endpoint is a specific URL your backend exposes to perform an action. Every endpoint is a digital door into your server:

  • GET /api/users/123 → Fetch profile data for user 123
  • POST /api/orders → Create a new order
  • DELETE /api/posts/45 → Remove post 45

🕵️ How Attackers Discover Your Exposed Endpoints

Before launching exploits, penetration testers and hackers map out every door in your API using 4 primary reconnaissance methods:

Reconnaissance Method How Attackers Execute It Enterprise Prevention Rules
1. Exposed Swagger / OpenAPI Docs Checking default paths like /api-docs, /swagger, or /openapi.json. If left unprotected in production, attackers get your full API blueprint for free! Always require authentication on documentation routes or disable Swagger UI entirely in production environments.
2. Frontend JavaScript Source Maps Downloading your frontend React/Vue bundle (`bundle.js`) and searching for "/api/" strings to extract every internal route your frontend calls. Never assume an endpoint is "hidden" just because there is no UI button for it. Every backend endpoint must enforce strict server-side authorization!
3. Path Brute-Forcing Using automated tools like gobuster or ffuf with wordlists containing thousands of paths (e.g., /api/admin, /api/debug, /api/v1/internal). Implement strict rate limiting and remove all temporary/debug endpoints before merging code to production.
4. Google Dorking Searching Google search engine indexes using operators like site:yourcompany.com inurl:/api/ or intitle:"index of /api". Ensure proper `robots.txt` configuration and never expose sensitive directory listings.

Level 2: JWT Manipulation & Authentication Vulnerabilities

A JSON Web Token (JWT) consists of three Base64Url-encoded parts separated by dots: header.payload.signature. Because payloads are merely Base64 encoded (NOT encrypted), anyone can read them!

🚨 The Two Deadliest JWT Attacks

❌ Attack 1: Role Escalation & jwt.decode() Trap
const token = req.headers.authorization?.split(' ')[1]; // DANGER: jwt.decode() simply unpacks base64 without checking cryptographic signature! const user = jwt.decode(token); if (user.role === 'admin') { // Attacker modified Base64 payload to "role": "admin" and gets full access! }
✅ Defense: Always Enforce jwt.verify()
const jwt = require('jsonwebtoken'); function verifyToken(req, res, next) { const token = req.headers.authorization?.split(' ')[1]; try { // Cryptographic signature check! Throws error if payload was tampered or expired! const decoded = jwt.verify(token, process.env.JWT_SECRET); req.user = decoded; next(); } catch (err) { return res.status(401).json({ error: 'Invalid token signature' }); } }

Level 3: Authorization Giants — IDOR & Broken Access Control

Once authenticated, the server must enforce **Authorization**. The #1 most common API bug in modern web development is **IDOR (Insecure Direct Object Reference)**.

1. IDOR (Insecure Direct Object Reference / BOLA)

Imagine a user logs in and requests their order: GET /api/orders/1001 ✅. What happens if they change the URL number to GET /api/orders/1002? If the server returns someone else's order, you have an IDOR vulnerability! (Real Example: In 2019, Instagram suffered an IDOR where users could download anyone's private Stories simply by altering a URL ID!)

❌ WRONG: Fetches whatever ID is passed
// Server trusts the URL parameter blindly! const order = await db.query( 'SELECT * FROM orders WHERE id = ?', [req.params.id] );
✅ RIGHT: Always scope query to verified user
// Always enforce ownership matching verified JWT session! const order = await db.query( 'SELECT * FROM orders WHERE id = ? AND user_id = ?', [req.params.id, req.user.id] // req.user comes from verified token! );

2. Broken Function Level Access Control (BFLA)

When regular users can invoke administrative endpoints like DELETE /api/admin/users/55 because the backend forgot to check user roles.

✅ Defense: Role-Checking Middleware
function requireAdmin(req, res, next) { if (req.user.role !== 'admin') { return res.status(403).json({ error: 'Forbidden: Admin access required' }); } next(); } // Apply middleware strictly to administrative routes app.delete('/api/admin/users/:id', verifyToken, requireAdmin, deleteUser);

Level 4: SQL Injection & Mass Assignment

1. SQL Injection (SQLi) Simulator

When user input is concatenated into raw SQL queries (`' OR '1'='1`), attackers can log in as admins or execute destructive commands like '; DROP TABLE users; --.

2. Mass Assignment (Over-Posting)

When a developer passes req.body directly into an ORM or database update function without whitelisting fields, attackers can inject arbitrary attributes!

❌ WRONG: Spreading req.body directly
// Attacker sends { "name": "Bhoomi", "role": "admin", "balance": 999999 } await db.users.update(req.params.id, req.body); // Attacker just granted themselves admin privileges!
✅ RIGHT: Explicit Field Whitelisting (DTO)
// Destructure and whitelist only allowed fields! const { name, email } = req.body; await db.users.update(req.params.id, { name, email });

CSRF & CORS Rules

  • CSRF Defense: Set cookies to SameSite=Strict or Lax and require backend synchronizer tokens for state mutations.
  • CORS Defense: Never combine wildcard origin Access-Control-Allow-Origin: * with credentials enabled. Explicitly whitelist trusted origins.

Level 6: Broken Rate Limiting & Unrestricted Resource Consumption

Without rate limiting, attackers can perform brute-force password guesses (1 million attempts), drain expensive AI LLM token budgets (10,000 requests), or scrape your entire database.

✅ Enterprise Express Rate Limiting Implementation
const rateLimit = require('express-rate-limit'); // 1. Strict Limiter for Login Endpoints (Prevents brute-forcing) const loginLimiter = rateLimit({ windowMs: 15 * 60 * 1000, // 15 minutes window max: 5, // Limit each IP to 5 failed login attempts message: { error: 'Too many login attempts. Account temporarily locked for 15 minutes.' } }); // 2. API Key Limiter for B2B / AI Endpoints (Prevents budget drain) const apiLimiter = rateLimit({ windowMs: 1 * 60 * 1000, // 1 minute window max: 60, // 60 requests per minute per API Key keyGenerator: (req) => req.headers['x-api-key'] || req.ip }); app.post('/api/login', loginLimiter, handleLogin); app.post('/api/ai/generate', verifyToken, apiLimiter, handleAIGeneration);

📋 The Master API Security Checklist

Attack Vector Required Engineering Fix
IDOR / BOLAAlways scope DB queries to verified session ID (`WHERE id=? AND user_id=req.user.id`).
Broken Access ControlEnforce server-side role middleware (`requireAdmin`) on every protected route.
SQL InjectionAlways use parameterized queries (`$1`), never string concatenation.
Broken Rate LimitingApply `express-rate-limit` per IP/User/API-Key on all endpoints.
JWT ManipulationEnforce `jwt.verify()` with strong secrets; never use `jwt.decode()` for auth decisions.
Exposed EndpointsRequire authentication on docs (`/swagger`) and remove internal/debug paths before deploy.
Mass AssignmentWhitelist explicit allowed fields (`const { name } = req.body`), never spread `req.body`.
